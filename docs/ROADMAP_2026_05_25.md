## Workstream A — nrev-tables MCP plan (post-doc-correction)

### Goal

Wrap `nrev-tables-service` (staging today, prod soon) for agent use. **NOT** for building customer-facing dashboards — that path is the iframe pipeline in `nrev-ui-2/public/nrev-elite/dashboards/<id>/` per the [official playbook](https://github.com/nurturev/documentations/blob/main/nrev_tables/dashboard/dev_guide/dashboard_developer_playbook.md). MCP scope is **agent-using-tables-as-DB**: read, write, aggregate, query, debug.

### Proposed tool surface (20 tools)

#### Auth (3, mirrors `nrev-wf`)
- `set_jwt(jwt)` — in-memory only, never persisted
- `get_auth_status()` — returns email + last4 of JWT
- `set_api_base(env="prod"|"staging"|<custom>)` — defaults to prod once shipped

#### Table CRUD (4)
- `list_tables(name?, creators?, skip=0, limit=100)`
- `get_table(table_id)` — returns full schema incl. columns
- `create_table(name, columns=None)` — `columns` accepts `[{name, type, position?}]`; types: text|long_text|number|boolean|date|datetime|json
- `rename_table(table_id, new_name)`

#### Column ops (3)
- `add_column(table_id, name, type, position?)`
- `rename_column(table_id, column_id_or_name, new_name)`
- `reorder_column(table_id, column_id_or_name, position)`

#### Row ops (4)
- `list_rows(table_id, filter?, sort_by?, search?, columns?, skip=0, limit=100)` — `filter` accepts `{column: "<name or id>", operator, value}`; helper resolves names→ids; `columns` is a projection that returns name-keyed dicts
- `add_row(table_id, values, by_column_id=False)` — auto-translates names→ids
- `update_row(table_id, row_id, values, by_column_id=False)`
- `bulk_add_rows(table_id, rows, by_column_id=False, progress_every=10)` — loops single-row inserts. Documents per-call cost so caller picks CSV for large jobs.

#### CSV import (2)
- `import_csv(s3_uri, column_mapping, table_name?)` — wraps `POST /tables/csv/import`
- `append_csv(table_id, s3_uri, column_mapping, by_column_name=True)` — wraps `POST /tables/{id}/csv/append`; auto-resolves names→ids if `by_column_name=True`

#### M2 wrappers (4 — when platform ships them)
- `aggregate_table(table_id, group_by?, aggregations[], filter?, joins?)` — wraps `POST /tables/{id}/aggregate`
- `distinct_values(table_id, column, filter?)` — wraps `POST /tables/{id}/columns/{col}/distinct-values`
- `batch_read(reads[])` — wraps `POST /tables/batch-read` for first-paint round-trip collapsing
- `join_rows(left_table_id, joins[], filter?, columns?, limit?)` — wraps `POST /tables/{id}/join`

### Open questions

1. **Should we keep `fetch_all_rows` ergonomic helper** (auto-paginate with `limit=1000`)? Yes — caller-side pagination is annoying because of the `[100,500,1000]` enum constraint. Cap at `max_rows=10000` default.
2. **DELETE endpoints** — M1 will add them per HLD 01. Add `delete_row`, `delete_column`, `delete_table` (with `confirm=True` like nrev-wf) wrappers in v2 after they ship.
3. **JWT scope** — should we share a single MCP process between `nrev-wf` and `nrev-tables`, or run them as separate plugins? Recommend: separate plugins, separate JWT state per process. Lower blast radius.
4. **Versioning** — pin to OpenAPI schema version once platform exposes one. Today there's no `X-API-Version`; flag once it appears.

### Not building (yet)

- `publish_dashboard()` — would drop generated assets into `nrev-ui-2/public/nrev-elite/dashboards/<id>/` + bump `dashboard-versions.ts`. Needs HLD 05 (schema gen) + `nrev-ui-2` write access. v2.
- `latest_change_token()` — wait for the platform's change-feed plans.
- Client-side `summarize` / `join_tables` / `time_bucket` / `distinct_values` — superseded by M2 server endpoints.

## Workstream B — Magic Node / Custom Code reliability tests

### Why now

Yesterday's session burned cycles on a CC node that "couldn't generate 4 rows" and "couldn't add 4 columns to the pandas df." Two basic operations failed silently or with unhelpful errors. **The MCP must encode enough knowledge that Claude doesn't bounce on these patterns.** User direction: **prefer Magic Node over Custom Code** for transformations; CC only when MN can't express it.

### Hypothesized root causes (must verify, not assume)

Before designing tests, list what I *think* went wrong yesterday based on prior session findings. The ratifier should challenge each:

1. **Output schema not declared.** CC nodes have an explicit `output_columns` declaration. If you add 4 new columns to `df` but don't update `output_columns`, downstream sees the old schema → renders look broken. v0.2.20 Fix A enforced this for Pipedream parents but maybe not in all cases.
2. **`df` vs `df1` confusion.** Single-input CC nodes get `df`; multi-input get `df1, df2, ...`. If the code uses the wrong variable name, the sandbox throws `NameError`. v0.2.18 documented this but the docstring may not be loud enough.
3. **CC-with-parent schema clobber.** Probe 4 in the v0.2.21 investigation found that when CC is downstream of a Pipedream parent, its `outputs.columns_metadata` gets clobbered to the upstream's `[error, summary, payload]`. PUT-fixup doesn't work (platform-side constraint). Adding 4 columns to df may have technically succeeded but the schema was overwritten.
4. **Row-generation pattern unfamiliar.** Generating N rows from scratch is `pd.DataFrame({col: [v1, v2, ...]})` returned as the result. If the caller wrote `df.append(...)` or `df.loc[len(df)] = ...` it might've silently produced 0 or 1 row.
5. **Return value semantics.** Does CC use `return df` or implicit-last-expression? The sandbox likely uses one; using the other silently no-ops.
6. **`validate_custom_code` rubber-stamps.** The lint tool may emit only `E000` syntax errors and miss "your code returns nothing" or "your output_columns doesn't match what df has."

### Test matrix design

Organized so each test fails cleanly OR passes cleanly, with the failure mode visible in MCP output. **Run each test as a real partial_execute against staging or prod.** Capture: status, stdout/stderr, output schema observed vs declared, downstream visibility.

## Workstream C — Monitoring tools (3 use cases)

### Use cases from user

1. Find workflows interacting with a specific Google Sheet / Slack channel
2. Within that workflow, find the exact node that wrote the output to that sheet/channel
3. List important runs that happened today at the workflow level (not node level)

(Checkpointing is platform team's job — not in scope.)

### Proposed tools (4)

#### `find_workflows_using_resource(app, resource_identifier, deep=False)`

**What**: Scans all workflows in the tenant for nodes whose settings reference a given external resource.

**Signature**:
```python
find_workflows_using_resource(
    app: str,                # 'google_sheets' | 'slack' | 'gmail' | ...
    resource_identifier: str, # spreadsheet_id, channel_id, etc. — exact string match
    deep: bool = False        # if True, also scrape templated references like {{data.foo}} resolving to this
) -> list[dict]
```

**Returns**:
```python
[
  {
    "workflow_id": "wf_abc",
    "workflow_name": "Q2 enrichment",
    "matching_nodes": [
      {"node_id": "n_xyz", "type_id": "...add-single-row...", "name": "Append to Sheet"},
      ...
    ],
    "last_execution": {"execution_id": "...", "started_at": "...", "status": "completed"}
  },
  ...
]
```

**Implementation**: For each workflow → get_workflow → walk every block → for blocks whose `typeId` matches the app's known typeIds → check known settings keys for the resource identifier. Cache the typeId→app map. Cap workflows scanned at e.g. 200 with a `limit` param.

**Cost**: O(workflows × nodes_per_workflow) — many small GETs. ~10-30s for 100 workflows. Document this.

**Edge cases**:
- Templated references (`{{data.sheet_id}}`) — `deep=True` would have to look at upstream outputs, expensive
- Multiple sheets per node (rare but possible)
- Stale workflows that haven't run in months — should they show up? Yes, but flagged

#### `find_node_writing_to(workflow_id, app, resource_identifier)`

**What**: Targeted version of the above when you already know the workflow.

**Signature**:
```python
find_node_writing_to(
    workflow_id: str,
    app: str,
    resource_identifier: str
) -> dict | None
```

**Returns**:
```python
{
  "node_id": "n_xyz",
  "node_name": "Append to Sheet",
  "type_id": "...",
  "settings_path": "fields.spreadsheetId",   # where the match was found
  "last_executions": [
    {"execution_id": "...", "status": "completed", "started_at": "...", "rows_written": 12}
  ]
}
```

**Implementation**: get_workflow once → linear scan blocks → return first match (or all if `all=True`). Calls list_executions for that workflow scoped to that node for "last 5" runs.

#### `daily_run_digest(date=None, include_passive=False)`

**What**: Workflow-level summary of all executions on a given date.

**Signature**:
```python
daily_run_digest(
    date: str | None = None,   # ISO date; defaults to today
    include_passive: bool = False  # include workflows with 0 runs that day?
) -> dict
```

**Returns**:
```python
{
  "date": "2026-05-25",
  "summary": {
    "workflows_run": 14,
    "total_executions": 47,
    "successful": 41,
    "failed": 4,
    "in_flight": 2,
    "total_credits": 1234,
    "total_rows_processed": 50000,
  },
  "important": [
    # Sorted by importance: failures first, then high-volume, then long-running
    {
      "workflow_id": "wf_a", "workflow_name": "Daily enrichment",
      "runs": 1, "status": "failed",
      "first_error": "Node 'Apollo Search' failed: 429 rate limit",
      "credits": 0, "duration_ms": 23000,
      "importance_reason": "failed; ran in production daily-cadence"
    },
    {
      "workflow_id": "wf_b", "workflow_name": "Q2 outreach blast",
      "runs": 3, "status": "completed",
      "credits": 412, "duration_ms": 180000,
      "rows_processed": 8500,
      "importance_reason": "high credit spend"
    },
    ...
  ],
  "by_workflow": [  # full list, sorted by total credits desc
    {"workflow_id": "...", "name": "...", "runs": N, "successful": N, "failed": N, "credits": N},
    ...
  ]
}
```

**Implementation**: list_workflows → for each, list_executions filtered to date range → aggregate. Importance heuristic = (failed > 0 ? +100 : 0) + log(credits) + log(rows_processed).

**Cost**: One list_workflows + N list_executions. ~5-15s for 50 workflows. Cacheable per day.

#### `workflow_change_log(workflow_id, since=None)` — bonus

**What**: List of recent edits to a workflow (when did its structure last change?). Useful for "did anyone touch this before it broke?"

**Signature**:
```python
workflow_change_log(
    workflow_id: str,
    since: str | None = None  # ISO timestamp; defaults to 7d ago
) -> list[dict]
```

**Returns**: `[{timestamp, actor_email, change_type: "node_added|node_deleted|edge_changed|setting_modified", details}, ...]`

**Implementation**: depends on whether the platform exposes an audit log endpoint. If not, the MCP can't do this without sampling `get_workflow` snapshots over time. Drop if no platform support.

### Open questions

1. **App→typeId catalog**: where does it live? Currently inferred per-investigation. The MCP should bake in a static map (`'google_sheets' → ['pipedream/google-sheets:add-row', '...':list-rows', ...]`). How do we keep it fresh as new typeIds appear? Auto-build at startup via `list_node_definitions`?
2. **Resource-identifier extraction**: each Pipedream-wrapped action stores its target resource in different settings keys (`spreadsheetId`, `channel`, `sheet_id`, etc.). Need a small lookup table per app.
3. **Cross-tenant visibility**: `list_workflows` is tenant-scoped. Can the MCP find workflows owned by other tenant users? Yes (verified in earlier investigation — Sayanta saw Harsh's tables). But for monitoring, do we *want* to show another user's workflows in the digest? Yes — operations care about the whole tenant, not just self.
4. **Importance heuristic tunability**: should the user be able to provide their own weights? `daily_run_digest(weights={"failed": 200, "credits": 1, "duration_ms": 0.001})`. Defer to v2.

### Out of scope

- Real-time event stream (would need platform webhooks)
- Alerting / paging (separate concern; this is for ad-hoc query, not push)
- Per-row lineage ("which node produced this exact value") — too expensive and the v0.2.x MCP `trace_path` already covers structural lineage

## Cross-cutting questions for the ratifier

1. **Tool count creep**: nrev-wf is at 54; adding nrev-tables (20) + monitoring (4) takes us to 78. Is that healthy or do we need to consolidate? Should monitoring live in nrev-wf (where workflow context already is) or as a third plugin?
2. **JWT scope sharing**: today every MCP needs its own `set_jwt`. Worth a shared credential store? Or is per-process isolation the right call?
3. **Caching**: monitoring tools are read-heavy and idempotent. Should the MCP have an internal TTL cache (e.g. 60s on get_workflow, 5s on list_executions)? Or stay stateless?
4. **Cost telemetry**: every tool call has a latency + a credit cost. Should the MCP expose a `tool_call_log()` so callers can see what they spent in a session?
5. **Cross-MCP coordination**: if a monitoring tool says "workflow X failed", the natural next call is "show me the failing node's settings via nrev-wf." Should the monitoring tool surface `node_id` in a way that's drop-in for `nrev-wf get_node()`? Yes — design IDs to be interchangeable.
